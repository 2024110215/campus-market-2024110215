3. 项目结构清单

校园轻集市项目采用 Vue 3 + TypeScript + Vite 技术栈，遵循 SFC（单文件组件）开发规范，前后端分离架构。

```
seed/
├── db.json                    # JSON Server Mock 数据库
├── index.html                 # Vite 入口 HTML
├── package.json               # 项目依赖与脚本
├── tsconfig.json              # TypeScript 配置
├── vite.config.ts             # Vite 构建配置
│
└── src/
    ├── main.ts                # 应用入口：挂载 Pinia、Router、全局样式
    ├── App.vue                # 根组件
    ├── assets/
    │   └── main.css           # 全局样式变量与基础样式
    │
    ├── api/                   # 接口封装层
    │   ├── http.ts            # Axios 实例（baseURL、拦截器）
    │   ├── trade.ts           # 二手交易 API
    │   ├── lostFound.ts       # 失物招领 API
    │   ├── groupBuy.ts        # 拼单搭子 API
    │   ├── errand.ts          # 跑腿委托 API
    │   ├── favoriteApi.ts     # 收藏 API
    │   └── userApi.ts         # 用户 API
    │
    ├── router/
    │   └── index.ts           # Vue Router 路由配置（13 条路由）
    │
    ├── stores/                # Pinia 状态管理
    │   ├── user.ts            # 用户 Store
    │   └── favorite.ts        # 收藏 Store
    │
    ├── components/            # 通用组件
    │   ├── AppHeader.vue      # 顶部导航栏
    │   ├── AppNav.vue         # 导航链接
    │   ├── AppLayout.vue      # 公共布局容器
    │   ├── ItemCard.vue       # 通用列表卡片
    │   ├── FormField.vue      # 表单字段容器
    │   └── EmptyState.vue     # 空状态占位
    │
    ├── views/                 # 页面组件
    │   ├── HomeView.vue       # 首页
    │   ├── TradeView.vue      # 二手交易
    │   ├── LostFoundView.vue  # 失物招领
    │   ├── GroupBuyView.vue   # 拼单搭子
    │   ├── ErrandView.vue     # 跑腿委托
    │   ├── PublishView.vue    # 发布信息
    │   ├── ItemDetailView.vue # 商品详情
    │   ├── LoginView.vue      # 登录
    │   ├── RegisterView.vue   # 注册
    │   ├── MessageView.vue    # 消息中心
    │   ├── UserCenterView.vue # 个人中心
    │   └── NotFoundView.vue   # 404
    │
    └── types/
        └── index.ts           # TypeScript 类型定义
```

6. 改进方向

虽然项目已完整覆盖实训要求的全部核心功能，但仍存在以下可改进的方向：

功能层面
- 图片上传：目前商品没有图片展示，可集成文件上传功能，让商品信息更丰富
- 真实登录鉴权：当前登录为前端模拟，可接入真实后端 JWT 鉴权体系
- 搜索与筛选：列表页缺少搜索框和分类筛选功能，用户在海量数据中查找不便
- 分页加载：当数据量增大时，需要实现分页或无限滚动加载，避免一次性渲染过多 DOM
- 用户间通信：缺少私信、留言等买卖双方沟通渠道

技术层面
- 单元测试：引入 Vitest 编写组件和 Store 的单元测试，保障代码质量
- TypeScript 深度：进一步规范类型定义，减少 any 类型使用，提升类型安全
- 性能优化：组件按需加载（路由懒加载已实现，可进一步细化）、虚拟列表处理大数据
- 构建优化：配置 Vite 打包体积优化、CDN 资源分离
- 状态管理分层：引入 action 层处理复杂业务逻辑，保持 Store 文件简洁

工程层面
- CI/CD：配置 GitHub Actions 自动化构建和部署
- 代码规范：引入 ESLint + Prettier 统一团队代码风格
- Git 协作规范：分支管理策略、Code Review 流程
- 接口文档：接入 Swagger 或 Apifox，前后端接口规范管理

7. 实训总结

7 天内，我从零搭建了一个完整的 Vue 3 前端项目。回顾整个过程，每一天都有清晰的递进关系：

Day1 搭建环境、配置 Git、部署 AI 工具，建立规范化开发基础；Day2 创建页面骨架和路由系统，搭建应用的整体结构；Day3 接入 Mock 数据和接口层，让页面"动"起来；Day4 实现表单交互和数据提交，打通"展示→收集→提交"的完整链路；Day5 引入 Pinia 状态管理，解决跨页面共享状态的问题；Day6 优化交互细节、处理边界情况，提升用户体验；Day7 综合验收、整理文档、总结复盘。

这个递进过程本身就是前端工程化的缩影：先搭环境→再搭结构→接入数据→实现交互→管理状态→优化体验→验收交付。

最核心的收获是建立了"三层分离"的架构思维：
- 视图层（.vue 组件）只负责 UI 渲染和用户交互
- 状态层（Pinia Store）管理跨页面共享的业务状态
- 数据层（API 模块）封装所有后端接口调用逻辑

三层职责清晰、单向依赖，使得任何一个层的修改都不会波及其他层，这是项目可维护性的根本保证。

AI 辅助开发贯穿了整个实训过程。从 Day1 的环境配置指导、Day2 的路由代码生成、Day3 的接口封装、Day4 的表单校验逻辑、Day5 的 Pinia Store 结构，到 Day6 的 API 持久化改造方案，AI 在代码生成、方案设计、问题排查三个环节大幅提升了效率。但我也深刻认识到，AI 无法替代开发者做架构决策和业务判断——哪些数据进 Store、错误如何处理、数据结构如何设计，这些需要结合具体业务场景人工判断。"AI 提效、人做决策"，是这 7 天实训对 AI 辅助开发最深刻的理解。

校园轻集市项目已具备完整的校园信息发布、浏览、收藏、管理功能，是一个可演示、可扩展、可继续迭代的前端项目。7 天实训的结束，也是项目持续完善的起点。
