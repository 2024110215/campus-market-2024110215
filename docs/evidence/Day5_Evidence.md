Day5 Evidence - 状态管理与用户中心
1. 今日完成内容
今天完成了两个 Pinia Store 的创建与挂载，分别是 user.ts 和 favorite.ts。其中用户 Store 管理模拟的当前登录用户信息，包含用户名、学院、年级和头像等字段，并提供了更新资料和登录/登出的动作方法；收藏 Store 管理用户的收藏列表，支持添加、移除、切换收藏状态以及判断某项是否已被收藏。在此基础上，我将用户 Store 接入到导航栏组件中，使页面顶部能够显示当前用户的名称和头像；将用户 Store 应用到发布页面，使新发布的业务数据自动关联当前用户作为发布人；在二手交易列表页的每个卡片上增加了收藏按钮，通过收藏 Store 实现点击切换收藏状态；最后完善了个人中心页面，集中展示用户资料和我的收藏列表。此外，我还对消息中心页面做了简单的静态占位，为后续交互优化预留空间。所有改动已通过 Git 提交。

2. Store 设计说明
用户 Store（src/stores/user.ts）
管理内容：当前登录用户的身份信息及登录状态。
主要状态：isLoggedIn（布尔值，表示是否已登录）、currentUser（对象，包含 name、college、grade、avatar 等字段）。
主要方法：updateProfile(payload) 用于更新用户资料字段；login(userInfo) 用于模拟登录，将用户信息写入 state 并将 isLoggedIn 置为 true；logout() 用于登出，清空用户信息并置登录状态为 false。

收藏 Store（src/stores/favorite.ts）
管理内容：用户收藏的项目列表，以及相关的操作逻辑。
主要状态：favorites（数组，每个元素包含 type 和 id，用来唯一标识一个收藏项）。
主要方法：addFavorite(type, id) 向列表中添加新收藏；removeFavorite(type, id) 移除指定收藏；toggleFavorite(type, id) 切换收藏状态（若已存在则移除，否则添加）；isFavorite(type, id) 返回布尔值，用于判断某个项目是否已被收藏。

这两个 Store 的职责边界清晰，用户 Store 只负责与“人”相关的信息，收藏 Store 只负责“收藏关系”，互不干扰，便于后续扩展。

3. 状态边界说明
在本次实训中，我遵循“仅跨页面共享的状态才放入 Store”的原则，对数据进行了明确划分：

放入 Store 的数据

当前用户信息：导航栏、发布页、个人中心等多个组件都需要读取，且用户资料可能在个人中心被修改，修改后导航栏应同步更新，因此放入 userStore 是最合适的选择。

收藏列表：列表页的收藏按钮需要读取“是否已收藏”状态，个人中心需要展示所有收藏项，两处共享同一份数据，因此放入 favoriteStore 统一管理。

不放入 Store 的数据

各列表页面从 JSON Server 获取的业务数据（如二手商品列表、失物招领列表等）：这些数据只在本页面渲染使用，且不同页面的数据结构不同，没有跨页面共享的必要，因此留在组件内部的 ref 或 reactive 中，通过接口请求获得。

发布表单中的临时输入内容（如标题、价格、描述等）：这些数据仅在当前发布页面使用，提交后即清空，无需持久化到全局状态。

表单校验错误信息：属于当前表单的临时反馈，只影响本页面 UI，完全不适合放入 Store。

通过这样的边界划分，Store 中只保留了必要的共享状态，避免了全局状态泛滥导致难以维护的问题。

4. 页面使用记录
AppHeader.vue：引入 userStore，在顶部右侧显示 userStore.currentUser.name 和头像，并在用户未登录时显示“游客”占位。

PublishView.vue：在提交表单数据时，从 userStore.currentUser.name 读取发布人姓名，赋值给 publisher 字段，替代了 Day4 中写死的固定值。

TradeView.vue（二手交易列表页）：引入 favoriteStore，在每个商品卡片底部增加收藏按钮，通过 favoriteStore.isFavorite('trade', item.id) 控制按钮文字（“收藏”/“已收藏”），点击时调用 toggleFavorite 方法。

UserCenterView.vue：同时引入两个 Store。顶部展示 userStore 中的用户资料卡片；下方“我的收藏”区域直接渲染 favoriteStore.favorites 数组，并调用相应业务接口（如获取二手商品详情）来展示收藏项的名称和缩略图（为简化，本次仅展示收藏的 id 和类型，进阶时再完善）。

MessageView.vue：仅作为静态页面占位，未使用 Store，为 Day6 预留。

5. AI 协作记录
本次实训我使用了豆包 AI 辅助生成代码。核心提示词如下：

“我使用 Vue 3 + Pinia 开发一个校园信息平台，需要创建两个 Store：一个管理用户信息（模拟登录），另一个管理收藏功能（前端内存存储）。请分别给出 user.ts 和 favorite.ts 的代码，使用 options 写法，包含 state、getters、actions。不需要连接后端，只做模拟数据。”

AI 生成了两个完整的 Store 文件，以及如何在组件中引入使用的示例代码。生成的内容基本符合要求，但也存在以下不合理之处：

AI 在用户 Store 中加入了 login 和 register 方法，并模拟了异步请求和 JWT Token 的存储逻辑，这显然超出了 Day5 的范围。

收藏 Store 中，AI 使用了 localStorage 进行持久化，虽然实用，但手册明确要求“暂时只保存在内存中”，以便学生理解状态刷新丢失的特性。

AI 生成的部分 getter 命名过于复杂，例如 favoriteCountByType，而本日只需简单的总数即可。

6. 人工调整内容
针对 AI 生成的内容，我做了以下人工修改：

删除了用户 Store 中的 register 方法和所有 JWT 相关逻辑，仅保留模拟的 login（直接写入用户信息）和 logout。

移除了收藏 Store 中的 localStorage 读写操作，改为纯内存数组，并在 toggleFavorite 中直接操作数组，不持久化。

简化了 getter，只保留一个 totalFavorites 用于计算收藏总数，其余复杂 getter 全部删除。

将 AI 示例中使用的 <script setup> 组合式 API 写法统一改为选项式（因为我们的项目主要使用选项式），并调整了组件内引入 Store 的方式为 import { useUserStore } from '@/stores/user' 并直接使用。

在个人中心页面，AI 只给出了“我的收藏”列表的简单占位，我补充了通过 favorites 数组反向查询业务数据的逻辑（虽然仅展示 id，但结构已搭好）。

修正了 AI 在 isFavorite 方法中的判断条件，原代码使用 findIndex 但未处理 -1 情况，我改用 some 更简洁。

8. 遇到的问题与解决方法
问题一：useFavoriteStore 在组件中调用时报错“getActivePinia was called with no active Pinia”
原因是我在 main.ts 中忘记挂载 Pinia 插件。解决：在 app.use(createPinia()) 后重新启动项目，问题消失。

问题二：刷新页面后收藏数据丢失
这是预期行为，因为当前收藏 Store 仅保存在内存中。我在证据卡中已说明，并在代码注释中标注“后续可接入 localStorage 或后端接口”。没有做额外处理，因为 Day5 的目标是理解状态共享而非持久化。

问题三：AI 生成的 toggleFavorite 方法中，使用了 if (this.isFavorite(type, id)) 但该方法在 action 内部调用时 this 指向问题
我将 isFavorite 改为普通函数并在 toggleFavorite 中直接使用 this.favorites.some(...)，避免了 this 指向混乱，同时使代码更加直观。

9. 今日反思
通过今天对 Pinia 状态管理的学习与实践，我深刻体会到在前端多页面应用中，统一的状态管理机制是多么重要。在没有引入 Pinia 之前，导航栏、发布页和个人中心之间的用户信息只能通过 props 逐层传递，或者直接从 localStorage 读取，这不仅代码冗余，而且难以保证数据的一致性。现在，所有共享状态都收敛到 Store 中，任何组件都可以直接读取或修改，并且借助 Pinia 的响应式特性，一处变化处处更新，极大提升了开发效率和维护性。用户中心作为典型的多数据源页面，同时依赖用户信息和收藏状态，用 Pinia 管理后，页面逻辑变得非常清晰——我只需要从两个 Store 中取数据，剩下的只是渲染问题。同时，我也学会了状态边界的判断：不是所有数据都塞进 Store，只有真正需要跨页面共享的才放进去，这是良好的设计习惯。今天的 AI 协作也让我警惕，AI 虽然能快速生成代码，但它可能会加入不必要的功能（如 JWT、持久化），我必须结合业务需求进行人工裁剪，否则会偏离实训目标。今后我会继续用“AI 生成 + 人工审查”的模式，但会更加注重对状态边界的把控，让状态管理真正服务于项目结构，而不是滥用全局状态。
