Day6 Evidence - 交互优化与体验完善

1. 今日完成内容

今天围绕交互优化与用户体验完善展开工作，核心目标是将 Day5 基于内存的收藏系统改造为后端 API 持久化方案，同时补齐列表页的空状态、加载状态和错误处理等边界场景，打磨 UI 交互细节。

收藏系统接入后端 API：创建收藏专用 API 模块 src/api/favoriteApi.ts，封装 getFavorites、addFavorite、removeFavorite 等接口方法，对接 JSON Server 的 favorites 数据集合。同时改造 Pinia 的 favorite Store，将原有的纯内存操作改为请求后端接口读写数据，实现收藏状态的跨页面持久化——刷新页面后收藏数据不会丢失。

页面状态完善：为四大业务列表页面（二手交易、失物招领、拼单搭子、跑腿委托）补充空状态展示（EmptyState 组件）和加载中状态（loading 提示），确保用户在数据为空或网络延迟时获得清晰的界面反馈。

错误处理统一：封装统一的错误提示逻辑，在 API 请求失败时给出用户可读的错误消息，替代浏览器默认的 alert 弹窗风格，使错误提示融入页面 UI 体系。

UI 细节打磨：统一调整列表卡片悬停效果、按钮反馈动画、页面间距和排版一致性，提升整体视觉质感。

2. 收藏 API 设计说明

收藏功能从纯前端内存存储升级为后端持久化方案，涉及以下层面的改动：

API 层（src/api/favoriteApi.ts）
封装四个接口方法：
- getFavorites(userId)：根据用户 ID 请求该用户的收藏列表
- addFavorite(data)：向服务端新增一条收藏记录
- removeFavorite(id)：根据收藏记录 ID 删除指定收藏
- isFavorited(userId, type, itemId)：查询指定项目是否已被当前用户收藏

接口统一使用 Axios 实例，baseURL 指向 JSON Server（http://localhost:3001），请求/响应走统一的拦截器逻辑。

Store 层（src/stores/favorite.ts）
改造后的 favorite Store 不再直接操作内存数组，而是通过调用 API 方法读写数据：
- 初始化时调用 getFavorites 拉取服务端数据填充 favorites 列表
- addFavorite 调用 API 请求写入后端，成功后再更新本地状态
- removeFavorite 调用 API 请求删除后端数据，成功后再更新本地状态
- 所有接口请求均包含 try/catch 错误处理，请求失败时回滚本地状态并提示用户

通过这一改造，收藏数据存入 JSON Server 的 db.json 文件中，页面刷新后重新请求接口恢复状态，实现了真正的数据持久化。

3. 页面状态完善

在 Day3 至 Day5 的开发中，四个业务列表页面仅在数据正常返回时渲染列表，缺少对边界情况的处理。今日对以下三种状态进行补充：

空状态
当列表数据为空数组时，不再显示空白页面，而是引入 EmptyState 组件展示提示信息和图标，引导用户发布内容。
- 二手交易列表空时显示："还没有商品，快去发布第一个吧"
- 失物招领列表空时显示："暂无失物/招领信息"
- 拼单搭子列表空时显示："还没有拼单活动"
- 跑腿委托列表空时显示："还没有跑腿任务"

加载状态
所有列表页面在 onMounted 中请求数据时，设置 loading 变量控制加载提示的显示。在数据返回前，页面中央显示"加载中..."或旋转加载图标，避免用户误以为页面卡顿。

错误状态
当接口请求异常（网络断开、JSON Server 未启动等），页面显示错误提示，并提供重试按钮或返回首页的操作入口，提升容错能力。

此外，在 PublishView.vue 的发布表单提交和 TradeView.vue 的收藏操作中，也统一加入了 loading 加载状态处理，按钮在请求期间显示"提交中...""处理中..."并禁用点击，防止用户重复操作。

4. 错误处理统一

此前项目中各页面错误提示方式不统一：有的用 alert()，有的在控制台输出，有的直接忽略。今日统一封装了以下错误处理方案：

统一错误捕获
在 API 层（http.ts）的 Axios 响应拦截器中，统一捕获 HTTP 错误状态码：
- 500+：提示"服务器错误，请稍后重试"
- 404：提示"请求的资源不存在"
- 网络断开：提示"网络连接失败，请检查服务是否启动"

Store 层错误隔离
在每个 Store action 中使用 try/catch 包裹业务逻辑：
- 成功时正常更新本地状态
- 失败时保持原有状态不变，并将错误信息通过返回值或回调传递到组件层展示

页面反馈
列表页错误时，页面保留上一次成功加载的数据（如果有），不因请求失败而清空列表；同时页面顶部显示一条可关闭的 Toast 提示错误信息。

7. 遇到的问题与解决方法

问题一：Pinia Store 中调用 API 请求时，this 指向错误
在 favorite Store 的 action 中调用 this.favorites 访问 state，但 async 函数中的 this 在某些场景下指向 undefined。排查后发现是 Store 定义使用了箭头函数导致的。解决方法：将 action 全部改为普通函数声明，确保 this 绑定正确。

问题二：JSON Server 不支持关联查询
收藏列表中需要展示收藏项目的标题、描述等信息，但 JSON Server 的 /favorites 接口只返回了 type 和 itemId 字段。如果前端需要展示项目标题，必须再逐一请求各业务接口查询详情。为解决这一问题，我调整数据结构：在添加收藏时，将业务数据的关键展示字段（title、description、location）冗余存储在 favorites 集合中，这样列表渲染时无需额外请求，提升页面加载速度。

问题三：刷新页面后收藏状态短暂空白
Store 初始化时需要通过异步请求拉取收藏列表，在此期间页面渲染时 isFavorite 判断始终返回 false，导致用户看到收藏按钮处于"未收藏"状态，随后才更新为正确状态。这导致短暂的界面闪烁。解决方法：在 Store 中添加一个 loaded 状态标记，初始化时默认为 false，API 请求返回后置为 true。组件中根据 loaded 状态控制渲染时机，避免状态闪烁。

8. 今日反思

通过今天的开发，我深刻体会到前端开发中"边界情况"的重要性。功能的核心路径往往很容易实现（收藏的添加/移除），但真正决定用户体验的是各种边界情况的处理：数据为空时展示什么？网络延迟时用户看到什么？请求失败时如何提示又不破坏现有内容？

将收藏系统从内存存储升级为后端持久化，让我理解了前端状态管理的一个关键原则：Store 是"缓存"而非"数据源"。真正可靠的数据在后端（JSON Server），前端 Store 只是后端数据的一份快照。页面加载时先从后端拉取数据填充 Store，用户操作时先请求后端，成功后再更新 Store——这个"后端可靠、前端缓存"的思维转变，是前端开发从入门到进阶的重要一步。

同时，在错误处理统一的过程中，我认识到好的错误处理应该是"有状态"的：不是简单地弹一个 alert，而是根据错误类型和当前页面状态给出恰当的反馈。列表页已经有数据时，请求失败应保留旧数据；发布页提交失败时，应保留用户已填写的内容允许重试——这些细节决定了应用的专业程度。

今天的 AI 辅助重点用在了 API 封装和 Store 改造方案上，但具体到错误处理逻辑、状态闪烁修复、数据结构设计等需要结合业务场景判断的地方，仍然需要人工介入决策。这进一步验证了"AI 提效、人做决策"的协作模式。
