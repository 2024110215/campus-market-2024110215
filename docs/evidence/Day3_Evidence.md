二、Mock数据结构说明
我结合各业务的页面展示需求与核心使用场景，设计对应数据集合与字段，适配页面渲染、状态展示、信息筛选等功能，具体结构和用途如下：
我结合各业务的页面展示需求与核心使用场景，设计对应数据集合与字段，适配页面渲染、状态展示、信息筛选等功能，四大模块具体数据结构与页面用途如下：
1. trades（二手交易）
核心字段：title、price、category、condition、location、publisher、publishTime、status、description 
页面用途：用于渲染二手商品列表，完整展示商品价格、新旧成色、分类、发布地点、发布时间以及商品描述等交易核心参数，为校园用户选购二手商品提供详细参考信息。
2. lostFounds（失物招领）
核心字段：title、type、itemName、location、eventTime、contact、status、description 
页面用途：统一承载校园寻物和招领两类信息，依靠type字段区分业务类型，页面可根据类型渲染不同状态标签，实现信息分类展示，让用户快速区分寻物、招领内容。
3. groupBuys（拼单搭子）
核心字段：title、type、targetCount、currentCount、deadline、location、publisher、status、description
页面用途：展示校园购物拼单、活动搭子等组队信息，通过目标人数和当前参与人数计算拼单进度，结合截止时间直观展示组队状态，方便用户判断是否参与拼单。
4. errands（跑腿委托）
核心字段：title、taskType、reward、from、to、deadline、publisher、status、description 
页面用途：展示校园各类跑腿代办任务，清晰呈现任务类型、酬劳、起止地点、截止时间等关键信息，为接单用户提供完整的任务参考依据。
三、数据结构设计思路
本次所有数据字段均基于校园业务真实使用场景设计，贴合用户浏览、使用的核心需求，针对性适配各业务逻辑：
- 二手交易核心字段设计：价格（price）和物品成色（condition）是二手交易的核心决策依据。校园用户选购二手商品时，会优先关注价格是否合适、物品新旧程度是否达标，这两个字段可以让用户快速判断商品性价比，贴合交易场景的核心需求。
- 失物招领核心字段设计：通过type字段区分“寻物”和“招领”两种场景。该业务页面需要同时承载两种不同类型的信息，通过字段做类型区分后，页面可以渲染不同的状态标签，实现信息分类展示，避免数据混杂、用户无法区分的问题。
- 拼单搭子核心字段设计：设置目标人数（targetCount）和当前人数（currentCount）。拼单、组队的核心逻辑是满员成团，通过两个字段可以实时计算拼单进度，在页面直观展示组队状态，让用户清晰了解拼单剩余名额和成团概率。
- 跑腿委托核心字段设计：设置起止地点（from、to）和酬劳（reward）字段。校园跑腿用户接单时，首要关注任务路线距离和对应酬劳，这两个核心字段可以帮助接单用户快速判断任务是否可行，贴合跑腿业务的接单逻辑。

六、 遇到的问题与解决方法
问题：db.json 文件格式错误，JSON Server 启动失败
现象：执行 json-server --watch db.json 后终端直接报错，服务无法正常启动，无法访问任何接口。
原因：编辑数据时末尾多写了逗号，JSON 语法不允许数组、对象最后一项带有多余逗号，文件格式不合法。
解决方法：删除对象 / 数组末尾多余的逗号，保证 JSON 标准格式，保存后重新运行启动命令，服务正常启动，全部接口可正常调用。
